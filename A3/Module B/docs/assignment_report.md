## 1. How Correctness of Operations Is Ensured

The backend uses layered controls to keep writes correct even under concurrent usage.

- Database transactions are used around critical write flows (cart updates, checkout, signup, delivery acceptance, profile updates).
- On error, operations rollback; on success, operations commit. This avoids partial state updates.
- ID generation for write-heavy entities is lock-backed through named DB locks instead of unsafe plain MAX(id)+1 logic.
- For high-contention paths, additional serialization was implemented:
  - Per-customer cart lock (`QB_CART:<customerID>`)
  - Per-order delivery-assignment lock (`QB_ORDER_ASSIGN:<orderID>`)
- Row-level cart reads now use `FOR UPDATE` in the cart upsert flow to prevent lost updates.
- Session transaction isolation is set to READ COMMITTED to reduce stale-read collisions for concurrent writers.
- Menu delete/restore endpoints are made idempotent (repeat calls do not corrupt state and are handled safely).
- Integrity constraints in schema (PK/FK/check constraints) provide consistency guards at database level.
- Audit logging records mutating operations for traceability and post-incident validation.

## 2. How Failures Are Handled

Failure handling is implemented using explicit rollback patterns and validation checks.

- Critical endpoint pattern:
  - Start operation
  - Validate inputs/state
  - Execute DB mutations
  - `commit()` on success
  - `rollback()` on exceptions
- Transactional signup and checkout paths reject invalid intermediate states and rollback entirely.
- Timeout/disconnect scenarios are tested to ensure no partial entities remain.
- Busy-contention conditions return conflict responses (HTTP 409) for safe client retry behavior.
- Soft-delete and restore operations are implemented with audit trails for recoverability.

## 3. How Multi-User Conflicts Are Handled

The system now includes explicit conflict controls for same-data parallel writes.

- Named locks serialize critical shared resources:
  - Same customer cart updates/checkout
  - Same order acceptance by multiple delivery partners
  - Lock-backed ID allocation in multiple creation flows
- Cart operations prevent interleaving writes that previously risked lost quantity updates.
- Delivery acceptance ensures one and only one successful assigner for a single order under race.
- Duplicate-key conflict handling and bounded retries are used where necessary.

## 5. Observations and Limitations

### Observations

- Correctness-focused controls significantly reduced write-path failures under load.
- Race-sensitive operations (signup, delivery acceptance, same-cart updates) now behave deterministically.
- Stress reliability is high (0.000% error in latest custom threading run and 0 failures in latest Locust run), but write-path latency still rises under pressure.

### Limitations

- Stress tests currently use synthetic traffic patterns and a bounded set of seeded users.
- Flask development server is not production-grade; production deployment should use Gunicorn/Uvicorn with worker tuning.
- P95 latency for write operations remains high under heavy concurrent load.
- Further optimization can include:
  - Additional DB indexes specific to hottest write/query patterns
  - Queue-based write smoothing for bursty workloads
  - More realistic mixed-role load models (customer + restaurant + delivery + admin simultaneously)

## 6. Conclusion

The backend now satisfies assignment requirements for correctness, failure handling, multi-user conflict safety, and experimental validation. ACID behavior is verified, race conditions are constrained using explicit locking and transactional controls, and stress behavior is stable with low error rate.
